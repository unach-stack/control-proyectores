================================================================================
  SISTEMA DE ENCARGADOS SEMANALES — CONTROL DE PROYECTORES UNACH
  Especificación funcional v1.0
================================================================================

CONTEXTO
--------
El sistema actual permite a cualquier estudiante autenticado solicitar un
proyector libremente. El problema es la falta de control: no se sabe quién
es responsable, si el responsable falla no hay protocolo, y los datos de
perfil (grado/grupo) no tienen verificación de autenticidad.

Esta especificación define dos sistemas complementarios:
  A) Verificación y corrección de perfil con supervisión admin
  B) Sistema de encargados semanales con flujo de aprobación y sustitución


================================================================================
  PARTE A — VERIFICACIÓN Y CORRECCIÓN DE PERFIL DE USUARIO
================================================================================

PROBLEMA
--------
- El estudiante registra su grado/grupo manualmente en GradeGroupModal al
  primer login, sin ninguna verificación externa.
- Si se equivocó (grado, grupo, turno, nombre) no tiene forma de corregirlo.
- Si se le permite editar libremente en cualquier momento, puede manipular
  sus datos para acceder a proyectores que no le corresponden.

SOLUCIÓN
--------
Flujo de solicitud de corrección de perfil:

  [Usuario] -> solicita correccion -> [Admin revisa] -> aprueba o rechaza
       |                                                      |
       |                                            si aprueba: se habilita
       |                                            ventana de edicion de
       |                                            24 horas para el user
       v
  Si rechaza: usuario recibe notificacion con motivo

REGLAS
------
- Un usuario solo puede tener UNA solicitud de corrección activa a la vez.
- El usuario no puede editar su perfil sin aprobación previa del admin.
- La ventana de edición expira en 24 horas desde la aprobación.
- Si el usuario no edita en 24 horas, la solicitud se cierra y debe
  repetir el proceso.
- El admin puede ver el historial de cambios de perfil de cada usuario.

MODELO DE DATOS NUEVO — SolicitudCorreccionPerfil
--------------------------------------------------
  usuarioId       ref User
  camposACorregir [String]        -- ej: ['grado', 'grupo', 'turno']
  valorActual     Object          -- snapshot actual del perfil
  justificacion   String          -- por qué se equivocó
  estado          pendiente | aprobado | rechazado | completado
  fechaAprobacion Date            -- cuando admin aprobó
  fechaExpiracion Date            -- +24h desde aprobación
  editadoPor      ref User (admin)
  createdAt       Date

================================================================================
  PARTE B — SISTEMA DE ENCARGADOS SEMANALES
================================================================================

CONCEPTO CENTRAL
----------------
Cada semana, UN estudiante por grupo queda designado como "Encargado".
Solo el Encargado activo puede acceder a RequestProjector para esa semana.
Si el Encargado no se presenta, puede ser sustituido por otro del mismo grupo
previa autorización del admin.

Los jueves y viernes de cada semana los estudiantes postulan para ser Encargado
de su grupo la semana siguiente. El admin revisa y designa uno.

ROLES
-----
  Usuario regular     — puede postularse como Encargado, puede solicitar
                        sustitucion si el encargado falla
  Encargado activo    — tiene acceso a RequestProjector esa semana
  Encargado provisional — acceso temporal si el titular no se presentó
  Admin               — aprueba postulaciones, verifica datos del grupo,
                        autoriza sustituciones, puede asignar directamente

CICLO SEMANAL
-------------

  JUEVES-VIERNES (semana actual)
  ├── Sistema abre ventana de postulaciones para la semana siguiente
  ├── Usuarios del mismo grado/grupo/turno pueden postularse
  └── Admin recibe lista de postulantes por grupo

  VIERNES (o antes del lunes siguiente)
  ├── Admin revisa postulantes: verifica grado/grupo en perfil
  ├── Admin designa UN Encargado por grupo para la semana siguiente
  │   (puede asignar directo si no hay postulantes)
  └── Sistema notifica al designado y cierra postulaciones del grupo

  LUNES-VIERNES (semana activa)
  ├── Solo el Encargado designado puede acceder a RequestProjector
  ├── Puede hacer solicitudes normales para su grupo esa semana
  └── Si no se presenta (no hace ninguna solicitud antes del miércoles 12:00):
      ├── Cualquier miembro del mismo grupo puede solicitar ser Encargado Provisional
      ├── Admin recibe alerta y puede aprobar al primero que solicite
      └── El Provisional tiene acceso hasta el viernes de esa semana

  DOMINGO
  └── Se cierran todas las encargadurías activas de la semana

REGLAS ANTI-TRAMPA
------------------
- Solo puede postularse quien tenga grado/grupo verificado o perfil original
  sin cambios recientes (menos de 7 días desde cambio de perfil).
- Un usuario no puede ser Encargado dos semanas seguidas del mismo grupo
  (rotación obligatoria). ✓ IMPLEMENTADO
- Si un grupo no tiene postulante para una semana, el admin puede asignar
  directamente o el grupo queda sin acceso esa semana. ✓ IMPLEMENTADO
- Solo los miembros del mismo grado+grupo+turno pueden postularse o
  solicitar sustitución para ese grupo. ✓ IMPLEMENTADO
- Un usuario solo puede ser Encargado de UN grupo a la vez. ✓ IMPLEMENTADO

MODELO DE DATOS NUEVO — Encargado
----------------------------------
  usuarioId       ref User
  grado           Number
  grupo           String
  turno           String
  semana          String        -- ej: "2026-W13" (año-semana ISO)
  tipo            titular | provisional
  estado          postulado | activo | inactivo | sustituido
  aprobadoPor     ref User (admin)
  fechaInicio     Date          -- lunes de la semana
  fechaFin        Date          -- viernes de la semana
  noSePresento    Boolean       -- marcado si no hizo solicitudes antes del miércoles
  sustituidoPor   ref User      -- si fue reemplazado
  createdAt       Date
